# 文档回写规则

> **这是单点定义文件**。定义在什么情况下必须更新哪些文档；保证仓库知识库与现实保持一致。

## 强制回写的四类场景

以下情况必须同步更新文档：

1. **改动改变长期边界、默认做法或验证路径**
   - 新增/删除/调整目录职责
   - 改变包管理器、构建工具、CI 流程
   - 调整入口装配顺序或应用壳层结构
2. **修复暴露出未来高概率重复出现的坑**
   - 调试/重构中发现的高频陷阱
   - 测试或构建的稳定易错点
3. **新增或拒绝某个会影响后续 AI 判断的长期决策**
   - 架构边界被确立或被调整
   - 某个方案被明确拒绝，且未来 AI 可能反复重新建议
   - **新增或修改仓库术语**（如"锚点 / 准入门禁 / 验证层 / 准入层 / verify 报告 / 验证证据 / Pre-Implementation Approval Gate / 行为不变量"等）必须在 [docs/CONTEXT.md](../CONTEXT.md) 登记或更新
4. **CI、部署、依赖、安全或环境行为发生变化**
   - 任何让旧验证方式失效的变更
   - 任何影响外部运行时行为的变更

## 文档分层与回写入口

仓库的文档职责建议按以下五层组织：

| 层级 | 位置 | 承担内容 |
|---|---|---|
| 1. 仓库级高频规则 | `AGENTS.md` | 跨任务通用、AI 会话启动时必须读到 |
| 2. AI 治理与工作流 | `docs/ai/` | 治理基线、入口、模板、清单、Runbook |
| 3. 单次任务设计 | `docs/specs/` | 中等及以上单次任务的目标、边界、技术决策和非目标 |
| 4. 实施计划 | `docs/plans/` | 把设计拆成可执行任务 |
| 5. 长期决策 | `docs/adr/` | 长期有效的技术决策，附理由与代价 |

## 决策矩阵

| 改动类型 | 回写目标 |
|---|---|
| 仓库级规则调整 | `AGENTS.md` |
| AI 治理或工作流变化 | `docs/ai/`（含本文档） |
| 单次任务的关键决策 | `docs/specs/<date>-<name>.md` |
| 可执行切片与验证 | `docs/plans/<date>-<name>.md` |
| 长期架构或流程决策 | `docs/adr/NNNN-<name>.md` |
| 高频执行陷阱 | `docs/ai/runbooks/` |
| 评审发现的新盲区 | `docs/ai/checklists/` |

## 回写时机

- **改动提交前**：在 PR 描述或最后一次 commit 之前完成必要的文档更新
- **修复关闭前**：在 [bugfix-brief.md](./templates/bugfix-brief.md) 的"需要保留的知识"中标注回写目标
- **评审中**：在 [checklists/review-checklist.md](./checklists/review-checklist.md) 的核心问题中检查文档回写是否遗漏

## 不回写的情况

- 临时调试产生的一次性发现
- 与本次任务直接相关的局部注释（应写在代码内而非文档）
- 私有配置或一次性脚本（除非它会被复用到未来任务）
