# 重构说明模板

## 目标

- 一句话说明这次重构的最终状态

## 级别

- 默认按 `L1` 处理；如果触及共享边界 / 入口装配 / 通用组件接口 / 脚手架约定 / workspace 结构，至少按 `L2` 处理（详见 [task-levels.md](../task-levels.md)）

## 行为不变量（必填）

- 哪些行为在重构后**必须保持不变**
- 哪些外部可见行为可以**允许**变化
- 行为不变意味着什么（接口签名 / 输出格式 / 性能 / 错误码等）

## 主战场文件清单

- 哪些文件是这次重构的主要改动点
- 哪些文件会被**间接**影响（import 链、类型推断、共享常量等）
- 哪些文件**不**应被改动（即使表面上看起来"顺手"）

## 行为未变验证

- 用什么命令 / 断言 / 截图来证明"行为未变"
- 与"功能开发"的差异：每一步都必须能展示"行为未变"的证据
- 建议至少包含：现有测试的回归 + 主流程的端到端验证

## 为什么不重写

- 为什么不直接重写？
- 重构（保留行为）vs 重写（可能改变行为）的边界

## 触及共享边界检查

- 这次重构是否触及任何共享边界（应用壳层、入口装配、根脚本、共享 utility、共享类型）
- 如触及，**必须**升级到 L2+，按 [ADR-0004](../../adr/0004-l2-spec-and-plan.md) 双写 spec + plan

## 风险面

- 哪些场景下"行为未变"可能不可证
- 哪些下游消费者可能受影响
- 哪些回归风险最难检测（性能、内存、并发等）

## 非目标

- 这次重构**不**做的事（即使是"看起来顺手"的重命名 / 抽取 / 简化）
- 不接受的"顺手扩大"清单
