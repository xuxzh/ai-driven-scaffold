# 验证基线

> **这是单点定义文件**。所有验证强度的收敛都按此文件分层；不指定具体命令，由各项目在 AGENTS.md 补全实际命令。

## 核心原则

按风险等级分层收敛，不要求所有改动都跑完整基线。选择验证的两条原则：

1. **必须足以暴露改动引入的回归**
2. **不应跑明显与当前切片无关的命令**（防止"全量跑一遍"造成的成本与噪声）

## 分层验证基线

| 级别 | 验证要求 |
|---|---|
| `L0` | 至少执行一个与改动直接相关的最小验证 |
| `L1` | 验证被修改的那一层（相关测试 + 局部 lint 或 typecheck） |
| `L2` | 覆盖静态检查、类型检查和受影响功能测试；主链路变化时补 E2E |
| `L3` | 在完整基线外，再人工确认验证范围是否足够 |

## 合并前验证强度（按改动类型）

| 改动类型 | 验证要求 |
|---|---|
| 纯文档改动 | 文档自检、占位符扫描、Git 状态确认；不要求运行完整验证 |
| 代码改动 | 运行与改动直接相关的单元测试，并按影响面补充 typecheck 和 lint |
| 工具链、依赖、构建、CI 或环境配置改动 | 运行完整验证，或在有明确原因时运行覆盖同等风险面的替代命令 |
| 验证失败且失败点与 diff 无关 | 标注为基线失败，说明失败文件和命令，由用户决定是否继续合并或先修复基线 |

## 默认完整基线

各项目应在根级 package/manifest 中定义一个 `verify` 命令，串联以下步骤：

1. 静态检查（lint）
2. 类型检查（typecheck；如不适用可省略）
3. 单元测试（test）
4. 端到端测试（test:e2e；如不适用可省略）
5. 构建（build）

具体命令由各项目按所选包管理器与构建工具决定，示例：

```text
<pm> lint
<pm> typecheck
<pm> test
<pm> test:e2e
<pm> build
```

其中 `<pm>` 是当前项目实际使用的包管理器（如 pnpm / npm / yarn / uv / cargo / go），应在 `AGENTS.md` 顶部"用户项目元信息"段落中标注。

## 验证证据的记录位置

- 任务完成时：在 PR/MR 描述或会话汇报中列出实际跑过的命令与结果
- 缺陷修复：在 `docs/specs/<date>-<fix>.md` 或 `AGENTS.md` 的变更说明中记录
- 长期回归追踪：在仓库 CI 的 artifacts 中保留（如 `test-results/`）
